# Deployment & Scalability Guide (10M Concurrent Users)

## Core Architectural Principles for Scale

To support 10 million concurrent users, this platform relies on an asynchronous, horizontally scalable, microservices architecture deployed on Kubernetes (EKS/GKE).

### 1. Edge & API Gateway (Load Balancing)
- **CDN**: Cloudflare or AWS CloudFront caches static media assets generated by the *Creative Agent*.
- **API Gateway**: Nginx Ingress or AWS API Gateway routes API traffic. Nodes are stateless, auto-scaling based on CPU/Memory usage.
- **WebSocket/SSE Connections**: Terminated at a specialized push-notification edge service to stream analytics updates and agent-generation progress without overwhelming backend APIs.

### 2. Microservices (Statelessness is Key)
- The services (`auth`, `orchestrator`, `publisher`, `analytics`) hold no state.
- **Auto-scaling**: Configure Horizontal Pod Autoscalers (HPA) targeting 60-70% CPU utilization.

### 3. Agent Orchestrator & LLM Gateway
- Calling LLM APIs (OpenAI, Anthropic) introduces significant latency.
- Orchestrator functions as a saga pattern orchestrator using **Kafka** or **Temporal.io**.
- E.g., The *Strategy Agent* publishes a `strategy_created` event to Kafka -> *Research Agent* picks it up, etc.
- **Fallback Routing**: Implement litellm/Portkey style routing. If OpenAI rate limits are hit, failover to Anthropic or Gemini seamlessly.

### 4. Database Layer Scale
- **PostgreSQL**: Used for relational data (Workspaces, Rules, Users). Deploy via Amazon Aurora (PostgreSQL Serverless V2) with read replicas. Write operations go to the primary node; read operations (analytics dashboards) go to replicas.
- **Redis (ElastiCache)**: Used for Session Management, Rate Limiting (crucial for 10M users), and caching frequent DB queries (like workspace configuration).
- **Kafka Service**: Event backbone. Kafka partitions ensure data order for social publishing while allowing thousands of consumer pods to process events concurrently.
- **Vector DB (Milvus/Pinecone)**: Used for the *Continuous Learning / SEO* loop. Embeds past high-performing posts using a separate async pipeline.

### 5. Social Publisher Worker Pool
- Consumes from a Kafka topic: `post_schedule_topic`.
- Workers check out jobs, process OAuth token rotation natively, call social media APIs, and write back to the DB upon success.
- Implement exponential backoff for API ratelimiting from Facebook/Twitter natively.

## Kubernetes Deployment Outline (Helm Structure)

```yaml
marketing-platform/
  charts/
    api-gateway/
    agent-orchestrator/
    publisher-workers/
    analytics-ingestion/
  values-prod.yaml
  values-staging.yaml
```

**Key Kubernetes configurations for `values-prod.yaml`**:
```yaml
agent-orchestrator:
  replicaCount: 150
  autoscaling:
    enabled: true
    minReplicas: 50
    maxReplicas: 500
    targetCPUUtilizationPercentage: 75
  resources:
    limits:
      cpu: "2"
      memory: "4Gi"
```
